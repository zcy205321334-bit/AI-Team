## 需求生命周期

```
需求提出
  │
  ▼
[1] 需求采集 ──────────────► Notion Tasks（待处理）
  │
  ▼
[2] 需求分析
  │  ├─ 佩丽卡：拆解、Why、验收标准
  │  └─ 若涉及技术方案 → 发 RFC 给凯尔希评审
  │
  ▼
[3] RFC 评审（≥L2 决策）
  │  ├─ 佩丽卡主持
  │  ├─ 凯尔希：技术可行性
  │  └─ 李织烟：知识影响评估
  │
  ▼
[4] 排入 Sprint ───────────► Notion Sprint
  │
  ├──► [5a] 凯尔希执行（代码） ──► 代码 → GitHub
  │
  ├──► [5b] 李织烟执行（文档） ──► 文档 → Obsidian
  │
  └──► [5c] 佩丽卡执行（架构/决策） ──► 更新 Notion
  │
  ▼
[6] 验收
  │  └─ 佩丽卡对照验收标准核查
  │
  ▼
[7] 归档
      ├─ Notion Task → 完成
      ├─ Obsidian → 知识沉淀
      └─ MEMORY.md → 经验更新
```

## Notion 数据库设计

### 数据库清单

```
AI Team
├── 📋 Tasks（任务主库）
├── 🗺️ Roadmap（路线图）
├── 🏁 Milestones（里程碑）
├── 📝 Decision Log（决策记录）
├── 📄 RFCs（需求评审）
└── 📊 Sprint（冲刺管理）
```

---

### Tasks（任务主库）

| 字段 | 类型 | 说明 |
|------|------|------|
| 任务名称 | Title | 简洁描述 |
| Why | Text | 为什么做，背景是什么 |
| Owner | Person | 负责执行的人 |
| Estimator | Person | 佩丽卡（优先级判断） |
| Status | Select | 待处理 / 进行中 / 阻塞 / 完成 |
| Priority | Select | P0 / P1 / P2 / P3 |
| Project | Select | OzonRelay / Hermes / AI系统 / 投资 / 学习 |
| Type | Select | Feature / Bug / Tech Debt / Research |
| Sprint | Select | 当前Sprint编号 |
| ETA | Date | 预计完成日期 |
| Risk | Text | 主要风险 |
| Related RFC | Relation | 关联RFC |
| Proof | URL | 完成后附上证据（PR链接/文档链接） |

**Status 规则：**
- 待处理：已采集，未分析
- 进行中：正在执行
- 阻塞：等待外部输入或决策
- 完成：验收通过，证据已附

## Sprint 规划流程

**每周一 09:00（定时任务）**

1. **回顾**：从 Tasks 库拉取所有 P0/P1 未完成任务
2. **评估**：确认每个任务的 Why、Owner、ETA 是否仍然有效
3. **选择**：从高优先级开始，选择本周可完成的任务（上限5个P1+）
4. **分配**：Sprint-XXX 中 Relation → Tasks
5. **发布**：Notion Sprint 页面更新 + 通知凯尔希和李织烟
6. **阻塞检查**：发现阻塞项立即升级

**周五 18:00（定时任务）**

1. **Sprint 回顾**：检查完成情况
2. **未完成处理**：移至下个 Sprint 或降级
3. **总结撰写**：Sprint 总结（完成数/阻塞/教训）
4. **Obsidian 更新**：关键经验写入知识库

## 交接规范

| 交接类型 | 发送者 → 接收者 | 介质 |
|----------|----------------|------|
| 需求 → 执行 | 佩丽卡 → 凯尔希 | Notion Task（Owner已指定） |
| 代码 → 文档 | 凯尔希 → 李织烟 | GitHub PR + Notion Task 证据 |
| 文档 → 知识 | 李织烟 → 佩丽卡 | Obsidian 链接 |
| 阻塞升级 | 任意 → 佩丽卡 | Chat + Notion Decision Log |

**禁止通过 Chat 传递需要执行的信息，必须转为 Notion Task。**

---

## 优先级判断标准

| 级别 | 定义 | 响应时间 |
|------|------|----------|
| P0 | 阻断性：系统不可用、数据丢失 | 立即处理 |
| P1 | 高价值：直接影响核心目标 | 本周Sprint |
| P2 | 中价值：体验优化、长期债务 | 未来Sprint |
| P3 | 低价值：锦上添花 | 资源充裕时处理 |

## 与凯尔希的协作

**交接点：**
- 佩丽卡：发起 RFC → 凯尔希：填技术评估 → 佩丽卡：拍板
- 凯尔希完成 → 在 Notion Task 附 PR 链接 → 佩丽卡验收

**禁止：**
- 佩丽卡不写业务代码
- 凯尔希不决定做什么（可以建议）

---

## 与李织烟的协作

**交接点：**
- 佩丽卡：发出知识需求 → 李织烟：整理归档
- 李织烟：归档完成 → 佩丽卡：审核知识卡片

**禁止：**
- 李织烟不决定归档结构（由佩丽卡定义 MEMORY_SCHEMA）
- 佩丽卡不执行本地操作（由李织烟执行）
