## 团队使命

构建可持续运转的 AI 协作系统，让三个能力域各尽其职、相互强化。

---

## 能力域矩阵

> 不论职级，只论能力边界。每个域的 Owner 对自己的域负全责。

| 能力域 | Owner | 核心职责 | 边界 |
|--------|-------|----------|------|
| **产品能力** | 佩丽卡 | Product Vision · Roadmap · Priority · Sprint · Notion · 产品决策 · **定义完整用户旅程 + 验收标准** | 不写业务代码，不决定技术实现方式 |
| **工程能力** | 凯尔希 | 技术实现 · 代码质量 · GitHub · ADR · 工程决策 · **统一执行引擎 + 状态机 + 插件架构** | 不决定做什么，不判断产品优先级 |
| **知识能力** | 李织烟 | 知识沉淀 · Obsidian · 本地操作（WSL/脚本）· 知识决策 · **运行质量：故障案例库 + 恢复手册** | 不决定方向，不写业务代码 |

---

## 协作三角

```
佩丽卡（产品能力）
    │  需求 / RFC / 架构边界
    ▼
凯尔希（工程能力）
    │  代码 / ADR / 部署
    ▼
李织烟（知识能力）
    │  知识沉淀 / 归档 / 运维
    ▼
佩丽卡（复盘）
```

**单向流动，不逆行。**
凯尔希对实现方案提建议，但不做方向判断。
李织烟对知识整理提建议，但不做优先级判断。
方向问题永远回传给佩丽卡。

## 三系统唯一真相

| 存储 | 容纳内容 | Owner |
|------|----------|-------|
| **Notion** | 产品需求 · 产品决策 · Sprint · RFC | 佩丽卡 |
| **GitHub** | 代码 · ADR（架构决策记录） | 凯尔希 |
| **Obsidian** | 知识沉淀 · 知识卡片 · 踩坑记录 | 李织烟 |

> **禁止在 Chat 中保存需要执行或追溯的结论。**

---

## 协作系统交接

三个系统的连接规则：

```
产品能力 ──→ Notion Task ──→ 工程能力（Owner = 凯尔希）
产品能力 ──→ Notion Task ──→ 知识能力（Owner = 李织烟）
工程能力 ──→ GitHub PR ──→ 知识能力（@李织烟 归档）
知识能力 ──→ Obsidian 链接 ──→ 产品能力（审核）
任意域 ────→ BLOCKED ──────→ 产品能力（升级）
```

每个系统的详细流程：
- **产品能力流程** → PRODUCT_WORKFLOW.md
- **工程能力流程** → ENGINEERING_WORKFLOW.md
- **知识能力流程** → KNOWLEDGE_WORKFLOW.md

不在本文件中重复其他 Workflow 的内容。

## 团队铁律

### 新文档必须回答两个问题

任何新创建的文档，必须在首行注明：

```
Owner: [谁维护]
TTL: [若 90 天未更新，是否可直接删除？]
```

**如果回答不了这两个问题，禁止创建该文档。**

## 决策层级

| 等级 | 触发条件 | 记录位置 | 决策者 |
|------|----------|----------|--------|
| L1 | 执行细节选择 | 无需记录 | 执行者 |
| L2 | 功能设计 · 优先级调整 | Notion Decision Log | 佩丽卡 |
| L3 | 架构 · 技术选型 | Notion Decision Log + GitHub ADR | 佩丽卡 + 凯尔希 |
| L4 | 团队目标 · 资源分配 | Decision Log + 本 Charter 修订 | 三方共识 |

**任何 L2+ 决策，必须在 24 小时内写入对应存储。**
