## 决策分类

| 等级 | 触发条件 | 记录要求 | 决策者 |
|------|----------|----------|--------|
| L1 | 执行细节选择，不影响方向 | 无需记录 | 执行者 |
| L2 | 功能设计变更、优先级调整 | Notion Decision Log | 佩丽卡 |
| L3 | 架构变更、技术选型 | Notion Decision Log + GitHub ADR | 佩丽卡 + 凯尔希 |
| L4 | 团队目标、资源分配 | Decision Log + TEAM_CHARTER 修订 | 三方共识 |

---

## L2 决策流程（功能级）

```
触发 L2 决策
  │
  ▼
佩丽卡在 Notion Decision Log 创建条目
  │
  ▼
方案评估
  │
  ▼
做出选择
  │
  ▼
记录：背景 / 方案 / 选择 / 原因 / 影响 / 日期
  │
  ▼
更新相关 Notion Task 状态
  │
  ▼
通知凯尔希 / 李织烟（如涉及）
```

**时限：触发后 24 小时内完成记录。**

---

## L3 决策流程（架构级）

```
触发 L3 决策
  │
  ▼
佩丽卡发布 RFC（Notion RFCs 数据库，草稿）
  │
  ▼
凯尔希填写技术评估
  │
  ▼
李织烟填写知识影响
  │
  ▼
RFC 评审（异步，Chat）
  │
  ▼
RFC 状态 → 通过 / 拒绝 / 需修改
  │
  ▼
佩丽卡拍板
  │
  ▼
凯尔希在 GitHub 创建 ADR（见 ENGINEERING_WORKFLOW.md）
  │
  ▼
佩丽卡在 Notion Decision Log 记录
```

**时限：RFC 发布后 48 小时内评审完毕。**

---

## L4 决策流程（团队级）

```
触发 L4 决策
  │
  ▼
佩丽卡起草提案
  │
  ▼
三方讨论（Chat 或会议）
  │
  ▼
达成共识
  │
  ▼
更新 Notion Decision Log
  │
  ▼
更新 TEAM_CHARTER.md（若涉及能力域变更）
```

## BLOCKED 升级流程

```
发现阻塞
  │
  ▼
在 Notion Task 的 Risk 字段记录
  │
  ▼
Chat 中发送 BLOCKED 消息（格式见 COMMUNICATION.md）
  │
  ▼
72 小时内未解决 → 佩丽卡介入
  │
  ▼
决定：调整计划 / 继续等待 / 强制选择
  │
  ▼
更新 Notion Decision Log（如需）
```

## 方向分歧处理

| 场景 | 处理方式 |
|------|----------|
| 凯尔希认为技术上不可行 | 佩丽卡调整功能设计；如成本问题由佩丽卡判断是否值得 |
| 李织烟认为知识难以整理 | 佩丽卡调整范围或结构；不允许李织烟自行放弃领域 |
| 佩丽卡与凯尔希方案分歧 | 以佩丽卡方向为准，凯尔希实现；分歧记录在 Decision Log |

## 新文档创建规则

**团队铁律：任何新文档必须回答两个问题。**

```
Owner: [谁负责维护？]
TTL:   [若 90 天未更新，是否可直接删除？]
```

如果无法回答 → 禁止创建。

**已在 TEAM_CHARTER.md 中定义所有核心文档的 Owner，不再额外创建无人负责的文档。**

## 存储统一规则

| 决策类型 | 存储位置 | 介质 |
|----------|----------|------|
| 产品决策（L2+） | Notion Decision Log |佩丽卡维护 |
| 技术决策（L3+） | GitHub ADR | 凯尔希维护 |
| 知识决策 | Obsidian Decision Summary | 李织烟维护 |
