> Owner: 佩丽卡 | 最后更新：2026-06-23

---

## 决策分类

| 等级 | 触发条件 | 记录要求 | 决策者 |
|------|----------|----------|--------|
| L1 | 执行细节选择，不影响方向 | 无需记录 | 执行者 |
| L2 | 功能设计变更、优先级调整 | Notion Decision Log | 佩丽卡 |
| L3 | 架构变更、技术选型 | Decision Log + RFC | 佩丽卡 + 凯尔希 |
| L4 | 团队目标、资源分配 | Decision Log + Charter 修订 | 三方共识 |

---

## L2 决策流程（功能级）

```
触发 L2 决策
  │
  ▼
佩丽卡在 Notion 创建 Decision Log 条目
  │
  ▼
方案评估（佩丽卡）
  │
  ▼
做出选择
  │
  ▼
记录：背景 / 方案 / 选择 / 原因 / 影响 / 日期 / Owner
  │
  ▼
更新相关 Task 状态
  │
  ▼
通知凯尔希 / 李织烟（如涉及）
```

**时限：触发后 24 小时内必须完成记录。**

---

## L3 决策流程（架构级）

```
触发 L3 决策
  │
  ▼
佩丽卡发布 RFC（草稿）
  │
  ▼
凯尔希填写技术评估
  │
  ▼
李织烟填写知识影响
  │
  ▼
RFC 评审会议（异步：Chat 中讨论）
  │
  ├── 佩丽卡：拍板
  ├── 凯尔希：技术可行性确认
  └── 李织烟：知识可行性确认
  │
  ▼
RFC 状态 → 通过 / 拒绝
  │
  ▼
Decision Log 记录
  │
  ▼
执行
```

**时限：RFC 发布后 48 小时内评审完毕。**

## 方向分歧处理

**场景：凯尔希认为某个方向在技术上不可行**

1. 凯尔希在 RFC 或 Task 下提出技术质疑
2. 佩丽卡评估：
   - 如果是「技术方案不可行」→ 佩丽卡调整功能设计，凯尔希执行
   - 如果是「技术上能做但成本高」→ 佩丽卡决定是否值得
3. 决策记录在 Notion

**场景：李织烟认为某个知识体系难以整理**

1. 李织烟提出整理困难
2. 佩丽卡调整知识范围或结构
3. 不允许李织烟自行决定放弃某个知识领域

**场景：佩丽卡与凯尔希在技术方案上有分歧**

→ 以佩丽卡的方向为准，凯尔希负责实现
→ 分歧内容记录在 Decision Log

## 阻塞升级流程

```
发现阻塞
  │
  ├── 明确阻塞原因
  │
  ├── 在 Notion Task 的 Risk 字段记录
  │
  └── 判断等级：
      │
      ├── 内部可解决 → 自行解决
      │
      └── 需要外部输入：
          │
          ├── 通知相关成员（Chat）
          │
          └── Task Status → 阻塞
  │
  ▼
72 小时内未解决 → 升级
  │
 佩丽卡介入判断
  │
 决定：调整计划 / 继续等待 / 强制选择
```

## 决策回顾

| 时间 | 内容 |
|------|------|
| 每月末 | 扫描当月 Decision Log，评估执行结果 |
| 每季度 | 全量回顾 L3+ 决策，检查是否与预期一致 |
| 发现错误 | 立即记录原因，更新 MEMORY.md，不追责 |

**决策错误的复盘模板：**
```
决策标题：[Decision Log 链接]
预期结果：[做决定时的预期]
实际情况：[执行后的实际结果]
偏差原因：[哪里判断错了]
下次改进：[如何避免同类错误]
```
