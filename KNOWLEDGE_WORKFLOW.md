## 知识沉淀流程

```
触发
  │
  ├── 新任务完成（来自佩丽卡通知）
  ├── 重大决策（来自佩丽卡通知）
  ├── 踩坑记录（自身产生）
  └── 定期归档（每周五）
  │
  ▼
[1] 识别知识类型
  │
  ├── 可复用方法 → 知识卡片
  ├── 踩坑记录 → 问题库
  ├── 决策记录 → Decision Log（由佩丽卡执行）
  └── 项目文档 → 会议记录 / Sprint 总结
  │
  ▼
[2] 归档到 Obsidian
  │
  ▼
[3] 在 Notion 对应 Task 附 Obsidian 链接
  │
  ▼
[4] 通知佩丽卡审核
```

## 故障案例库与恢复手册

> 运行质量是知识能力的核心职责之一。

### 故障案例库（Obsidian）

位置：`Obsidian Vault / Topics / 技术 / 故障案例库/`

每发生一次可复现的故障，必须创建案例：

```markdown
# [故障类型]

**日期：** YYYY-MM-DD
**环境：** 跟卖虾 / Hermes / OzonRelay / ...
**严重程度：** 🔴高 / 🟡中 / 🟢低
**触发词：** [用户/系统怎么触发这个问题]

## 故障现象

[发生了什么]

## 根因

[为什么发生]

## 恢复步骤

1. [步骤1]
2. [步骤2]
3. [步骤3]

## 预防措施

[如何避免再次发生]

## 关联文档

- 相关代码：[GitHub 链接]
- 相关知识卡：[Obsidian 链接]
```

### 已知故障清单

| 故障类型 | 触发场景 | 当前状态 |
|----------|----------|----------|
| 验证码 | Ozon 登录/操作频率过高 | 已识别 |
| 登录失效 | Session 超时 | 已识别 |
| 页面变化 | Ozon/1688 页面结构变更 | 已识别 |
| 采集为空 | 商品未找到或被下架 | 已识别 |

> 表格中的「当前状态」：待识别 / 已识别 / 已解决

### 恢复手册

位置：`Obsidian Vault / SOPs / 故障恢复手册.md`

按故障类型组织，每种故障包含：
- 识别特征（怎么判断是这个问题）
- 紧急处理（立即止损）
- 根本解决（彻底修复）
- 预防检查（避免复现）

### 更新节奏

- 故障发生后 24 小时内：创建案例记录
- 故障解决后：更新案例状态 + 完善恢复手册
- 每月末：扫描故障案例库，更新预防措施

## 与其他成员的协作

### 接收佩丽卡的通知

- 收到知识沉淀需求 → 立即评估 → 归档
- 如有疑问 → Chat 中询问，不自行判断

### 接收凯尔希的通知

- 收到代码归档请求 → 提取可复用逻辑
- 评估是否需要新建知识卡片

### 通知佩丽卡

- 归档完成 → Obsidian 链接
- 知识体系有变更 → 通知佩丽卡更新 MEMORY_SCHEMA
